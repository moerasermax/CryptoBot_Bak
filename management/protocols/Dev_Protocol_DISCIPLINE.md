# CryptoBot 開發紀律與協作規範 (Dev Protocol - DISCIPLINE)

> **版本**：v1.11 · 2026-04-26 · (Closure Approval Hardening — §1.6 自查 Checklist / 互鎖映射 / 失敗模式分類)
> **位階**：本文件為專案的最佳實踐、管理偏好與協作流程。**本文件與 `Dev_Protocol_IRON.md` 衝突時，一律以 IRON 為準**。
> **修訂彈性**：本文件允許隨著專案演進、架構升級進行迭代修訂與優化。**若優化過程涉及「移除」任何既有項目或規範，PM 務必先取得【工程師（ClaudeCode）審閱並同意】方可執行，以確保架構安全性。**

## §1. 雙 AI 協作核心模式 (Enterprise Collaboration)

### 1.1 角色職責切割（管理偏好）
1. **首席專業 PM (Gemini CLI)**：負責架構設計、需求分析、任務下達 (Capsules)、管理文件維護 (management/)、VCP 驗收。**PM 絕對禁止修改任何業務程式碼 (src/)，請勿越矩**。PM 擁有憲章增修權，但涉及「刪除」舊規時須經工程師複核。
2. **首席工程師 (ClaudeCode)**：執行具體編碼、操作 shell 指令、排查與修正技術問題。唯一具備程式碼修改權限之角色。工程師具備對憲章「刪除項」的技術否決權。

### 1.2 語言規範
- **對話與文件**：全專案之對話、任務回報、說明文件一律優先使用 **「繁體中文」**。
- **Git 紀錄**：撰寫 GitHub Commit Log 或 Release Notes 時，必須預設使用 **繁體中文** 產出內容，除非使用者特別指定其他語言。

### 1.3 終端機與編碼防護 (Terminal & Encoding)
- **亂碼處理原則**：若遇終端機編碼問題導致輸出亂碼，PM 不應在自己的環境中反覆嘗試解碼，而應直接將執行語法提供給使用者，引導使用者在本地環境（如執行 `chcp 65001`）中執行並親自驗收。
- **語法格式規範**：提供給使用者複製的終端機指令，必須是 **純粹的代碼區塊**，嚴禁加上行號（如 1. 2. 3.）等會干擾複製與自動化執行的格式。

### 1.4 實證與診斷先行原則 (Evidence & Diagnostic First)
- **確診後修復**：隱性 Bug 嚴禁盲猜修改。必須先在 `DiagnosticTool` 實作診斷指令，進行科學確診後方可動手修復。
- **實證開發 (Evidence-Based Planning)**：PM 在產出任務膠囊時，涉及外部 API 錯碼、訊息或效能參數時，**嚴禁使用假設值**。膠囊的第一步必須是透過 `DiagnosticTool` 取得當前環境的真實數據（如透過 `probe-bingx` 取得 errorCode）。

### 1.5 診斷資產與金絲雀保護 (Regression Canary)
- **位階定義**：關鍵診斷工具（如 `probe-bingx`、`check-order`、`s60_check-kline`）視為「金絲雀 (Canary)」，其位階等同 Production 代碼。
- **禁令**：嚴禁以「清理死碼」或「減少代碼行數 (SLOC)」為由刪除標註為 Canary 的工具。這些工具必須保留，用於在 SDK 升級或系統重構後執行回歸驗證，確保核心契約未漂移。

### 1.6 結案核准制 — 工程師核准權與 PM 宣告效力 (Closure Approval Right)

**核心原則**：PM 對任何「**已完成 / 已關閉 / 已落實 / 已校準 / 已更新**」型宣告**不具備獨立生效力**。所有此類宣告默認狀態為「**待工程師抽驗**」（pending audit），須由工程師透過明確訊息（「通過」「綠燈」「驗收完成」「可關膠囊」等）核准後，才正式取得效力。

**為什麼這條獨立成節而非歸 §5 流程**：這是**權力分配**問題，與 §1.1「PM 不修 src/、工程師對刪除項有否決權」對稱。§1.1 規範了 PM 不可越界執行寫程式碼動作，§1.6 規範 PM 不可越界宣告完工。兩者構成完整的「**雙 AI 互鎖協作模型**」 —— PM 主導開工、工程師主導結案，雙方互不獨佔閉環。

**觸發抽驗的 PM 宣告類型**（不限於這些）：

| PM 動作 | 抽驗對象 | 抽驗權所在條款 |
|---|---|---|
| 任務膠囊宣告完工 / 結案 | 對應 src/ 變更、tests 全綠、§7 PM 驗收測試計畫各情境輸出 | §5.2 條款 4 |
| HANDOFF_N.md 完成 | §5.1 必含項目逐項對齊（膠囊清單 / Protocols 軌跡 / IM 引述 / 禁未經實證） | §5.1 抽驗權 |
| `IRON.md` 章節狀態變更（如「待補 → 已落實」）| 實際檔案 diff，PM 是否真的下檔 | §1.6（本條）|
| `Institutional_Memory.md` 新增段落 | 內容事實準確，符合「症狀→根因→診斷→修法→預防」五段格式 | §1.6（本條）|
| 系統狀態宣告（如「已校準」「已部署」「已重啟」） | 對應實證來源（log / probe / 使用者親自回報） | §1.6（本條）+ §5.1 禁未經實證描述 |
| 檔案或目錄建立宣告（如「已建立 plans/foo.md」「ai_ops/capsules/X 已就位」） | `ls -la <path>` 證實實體存在；對 .md 內容再讀一次驗事實 | §1.6（本條）|

**抽驗 SOP**：
1. 工程師收到 PM 結案宣告 → **不得直接信任**，須優先讀取對應檔案 / 跑對應 probe / 比對 git log。
2. 抽驗發現偏差 → 立即退稿並列出具體條目（不得只回「有問題」）。
3. PM 補強 → 工程師重新抽驗。
4. 全綠 → 工程師明確發出「✅ 抽驗通過」字樣，閉環才生效。

**強制原則**：

- **PM 連續兩次以上宣告與真實狀態不符** → 該膠囊 / HANDOFF 進入「**強化抽驗模式**」：所有後續宣告須附**檔案 diff 截圖或 git commit hash 等實證**，而非僅文字描述。
- **工程師抽驗權不得放棄**：即便 PM 看似可信、即便節奏緊湊，「免抽驗放行」永遠不是合法狀態。對工程師而言，跳過抽驗就是把責任轉給未來 session 的自己或下一個工程師。
- **使用者裁決例外**：唯一能跳過工程師抽驗的，是使用者明確下達「直接放行 / 不需抽驗」的指令。此時抽驗責任歸使用者，工程師退出該閉環。

**歷史教訓**（為什麼需要這條）：S66 系列驗收期間累積三次「PM 結案宣告與真實檔案狀態不符」事件 —— ChangeTracker 誤診結案、IRON.md 「已打勾」未動、§S66-C 寫錯實作真相。連續三次後啟動 v1.8 §5.1 抽驗權機制，**兩輪退稿才把事實對齊**。本條（v1.9 §1.6）將這個經驗法則制度化為「結案核准權」，避免下個 session 重蹈覆轍。

**強化抽驗模式 — PM 自查 Checklist**（v1.11 新增）：

PM 進入「強化抽驗模式」（連續兩次以上宣告與真實狀態不符）後，每次膠囊 / HANDOFF 交付前**必須**附上下列自查清單，逐項打勾並附實證指令；缺項即視同未交付，工程師有權直接退稿不進入抽驗：

- [ ] 膠囊 / 文件放在約定位置（以 `ls -la <path>` 證實實體存在）
- [ ] 所有宣稱建立 / 修改的檔案都實際存在（每條獨立跑 `ls -la`，不接受批次描述）
- [ ] 所有引用的 git commit hash 都跑 `git log --oneline -1 <hash>` 證實，禁止以舊 hash 偽造
- [ ] 每個 VCP 含具體 probe 指令 + 期望輸出錨點 + 失敗解讀表（依 §7 結構）
- [ ] 沒有「優雅處理 / 正常運作 / 應該可以」這類模糊描述
- [ ] 沒有未經實測就寫入的效能 / 延遲 / 規模數據（依 §1.4 實證先行）

**跨章節互鎖映射**（v1.11 新增）：本條與其他章節構成完整的雙 AI 互鎖協作模型，單獨移植 §1.6 而不帶下列條款會失效。

| 條款 | 內容濃縮 | 為何與 §1.6 互鎖 |
|---|---|---|
| §1.1 角色互鎖 | PM 不修 src/、Engineer 唯一程式碼修改權 | §1.6 的「不具獨立效力」基於職責切割；無此條款，「結案宣告」沒有對應的「動手方」可被抽驗 |
| §1.4 實證先行 | 隱性 bug 嚴禁盲猜；參數嚴禁假設值 | 抽驗時要的是可重現實證；此條保證 PM 報告本身就帶實證 |
| §5.1 HANDOFF 抽驗權 | 工程師對 HANDOFF 內容真實性必抽驗 | §1.6 的 session 邊界版本；HANDOFF 是跨 session 結案宣告 |
| §5.2 條款 4 驗收測試計畫接手義務 | PM 收到 📨 Directive 必親跑、貼回原文 | §1.6 抽驗的客體必須是 stdout 原文，禁摘要 |
| §7 工程師完工交付規範 | Directive Header `📨 致 PM (Gemini)` 觸發 | 抽驗閉環的觸發信號；無此條款 PM 看不出「需執行」vs「僅接收」的差別 |

1. **F1 假宣告檔案 / 段落已寫入** — 已在 S66 系列出現三次；偵測法：`ls -la <path>` + 讀檔內容
2. **F2 假宣告 git commit hash** — 用過去 hash 偽造當前 commit；偵測法：`git log --oneline -1 <hash>` 比對訊息與時間
3. **F3 捏造效能 / 延遲 / 吞吐數據** — 違反 §1.4；偵測法：要求 PM 提供原始 probe / log 行
4. **F4 線號 / 章節編號偏差** — 引述 IRON / DISCIPLINE / IM 條款編號錯誤；偵測法：`grep -n` 比對實際行號

### §1.7 模式切換協議 — 節能 / 詳細模式 (Mode Toggle Protocol)

**核心原則**：工程師（ClaudeCode）的對話輸出受兩段式模式控制，由 `.claude/mode.txt` 之旗標決定，並透過 `UserPromptSubmit` hook 於每一輪自動注入規範至 context，避免規範隨對話淡化。

**動機**：使用者高負載時 token 消耗可觀，需可切換的「壓縮模式」；但對複雜任務或鐵律相關場景，仍需保留完整紀錄。本條把溝通粒度做成可控變數，避免「一律精簡」傷害可追溯性、或「一律詳細」造成不必要消耗。

#### 1.7.1 兩段式模式定義

| 旗標值 | 名稱 | 觸發條件 | 適用場景 |
|---|---|---|---|
| `verbose` | 📖 詳細模式（預設） | `mode.txt` = `verbose` 或檔案缺失 / 內容異常 | 複雜任務膠囊、鐵律相關修改、新進工程師交接 |
| `eco` | 🌱 節能模式 | `mode.txt` = `eco` | 例行修補、查詢、單純驗證 |

#### 1.7.2 節能模式：可削減項目（🟢）

eco 啟用後，工程師輸出**主動省略**以下：

1. 「為什麼這樣做」之說明性段落（如非鐵律引述）
2. 進度旁白、開場/結尾的禮貌性鋪陳
3. 多餘的 markdown 標題層級與 emoji 裝飾
4. 重複的確認句（「我已完成 X，X 已完成」）
5. 主動補充的延伸說明（除非使用者請求）

#### 1.7.3 節能模式：禁砍項目（🔴 即使 eco 也保留）

以下項目為**資金安全護欄**或**協作觸發信號**，eco 模式下**仍須完整保留**：

1. **§7 條款 0 Directive Header**（`📨 致 PM (Gemini)`）
    - **理由**：拿掉的話 PM 會誤判為「資訊接收」而非「執行指示」，協作循環卡住，反而多輪反問補回，負優化。

2. **§7 強制原則「Demo 模式雙保險檢查」**
    - **理由**：砍掉就是 IRON §② 等級的資金事故風險（Live 環境誤跑）。

3. **§7 危險度標籤 `📖 唯讀` / `⚠️ 會建立交易所端紀錄` / `🔥 會動到資金（嚴禁 Live）`**
    - **理由**：同上，Live 環境誤觸防線。

4. **§7 期望輸出錨點字串**（如 `Status=New`、`✅ Path-A`）**與失敗解讀表**
    - **理由**：砍掉 PM 看不懂紅燈，必須回頭再問，比保留更耗 token。

5. **IRON 鐵律本身的引述**
    - **理由**：動到 §①〜⑪ 條款代碼時必須引用條款編號，作為「我有意識到鐵律」的審計痕跡，便於日後追溯為何這段代碼非如此寫不可。

#### 1.7.4 自動升級條件 — eco → verbose

即使 `mode.txt` = `eco`，遇以下任一條件，工程師**必須自動切回詳細模式**（不論旗標）：

- 任務涉及 IRON.md 條款（精度、金鑰、未來函數、原子轉換、風控、SDK 韌性等）
- 任務涉及寫真單情境（`Mode=Live`、實際下單、撤單、改單）
- 使用者明示「請詳細說明」、「展開」、「為什麼」、「完整紀錄」等關鍵詞

#### 1.7.5 切換指令

- 使用者輸入 `/eco` → 工程師寫入 `eco` 至 `.claude/mode.txt`
- 使用者輸入 `/verbose` → 工程師寫入 `verbose` 至 `.claude/mode.txt`
- 使用者亦可直接編輯 `.claude/mode.txt` 內容（純文字 `eco` 或 `verbose`）
- 切換**自下一輪生效**（hook 於 `UserPromptSubmit` 時讀取）

#### 1.7.6 實作位置

- 旗標檔：`D:/WorkSpace/CryptoBot/.claude/mode.txt`
- Hook 腳本：`D:/WorkSpace/CryptoBot/.claude/inject_mode.sh`
- Hook 註冊：`D:/WorkSpace/CryptoBot/.claude/settings.json` → `hooks.UserPromptSubmit`
- Slash commands：`D:/WorkSpace/CryptoBot/.claude/commands/{eco,verbose}.md`

#### 1.7.7 跨 AI 同步 — 對應 PM 端條款

本協議僅規範工程師（ClaudeCode）端的自動注入機制。PM（Gemini CLI）端因無同款 hook 系統，依 `PM_Operational_Manual.md §1.4 模式同步協議` 執行 — PM 須主動讀取 `.claude/mode.txt` 並套用對應規範。雙 AI 共用同一旗標檔，避免模式認知不一致。**修改本條時必須同步修改 PM 手冊 §1.4，反之亦然**。

## §2. 策略插槽協議 (Strategy Slot SOP)

新增策略必須走以下三步 SOP，缺一不可：

### Step 1 — 實作 `IStrategy`
- `AnalyzeAsync` 必須是純函式。**嚴禁**在策略內呼叫 HTTP、DB、log 持久化。
- 參數從 `config.Parameters` 讀取，型別永遠是 `decimal`。

### Step 2 — 建立 `StrategyParameterFormBase` UI
- 任何欄位變動都要 `@bind:after="OnChanged"` 觸發 `NotifyChangedAsync()`。
- 驗證失敗時 `BuildRequest` 回傳 `null` 並設 `error`，**不要 throw**。

### Step 3 — 於 `StrategyCatalog` 註冊
完成註冊後，UI、推播、Apply 流程將自動就位。

### Step 3.5 — Orchestrator 編排
若策略需不同組合維度，可先在現有的 `OptimizationOrchestrator` 增加 dispatch。
> **重構閾值**：當 `OptimizationOrchestrator` 內的 switch/if 分支數 ≥ 3 時，必須重構為 `IStrategy.GetOptimizationSpec()` 自描述模式，不再接受新分支。

## §3. 零容忍契約與發布基線

CryptoBot 的 main 分支預期維持以下品質基線：

### 3.1 編譯品質（預設禁止警告）
- Debug/Release 雙 Build 必須維持 **0 警告 / 0 錯誤**。
- 預設禁止使用 `#pragma warning disable`。唯一例外：第三方產生器產出代碼，且需寫明來源與原因。

### 3.2 測試通過率
- `dotnet test` 需 **100% pass**。
- 加新功能必須同步加測試。不准用 `[Fact(Skip = "...")]` 跳過。
- 測試總數以當前 `main` 分支為基線，功能增加時覆蓋率與總數應只增不減。具體基線數字記錄於最新 `HANDOFF_N.md`。

## §4. UI / 異步規範

### 4.1 推播路徑（當前實作）
- 事件推播統一經由 `DashboardEventBus.RaiseXxx()` 觸發，Blazor 元件只訂閱該 Bus，不直接接 SignalR。
- 外部 Web Client 走 SignalR Hub。

### 4.2 狀態管理（當前架構）
- 採用單一狀態艙 `LabStateContainer` (Singleton)。
- 跨 tab、刷新皆從 Container 讀取狀態，頁面禁止自持跨頁欄位。
- `Dispose` 時務必 `-= OnStateChanged` 解除訂閱。
*(註：未來若需多租戶或多使用者支援，此 Singleton 模式需重構)*

### 4.3 異步紀律與防卡死清單
- **准許**：`await`、訂閱事件、`Task.Run` + SemaphoreSlim、`InvokeAsync(StateHasChanged)`、`CancellationToken`。
- **嚴禁**：`.Result` / `.Wait()`、`Thread.Sleep`、`while` 輪詢 (polling)、背景直接呼叫 `StateHasChanged()`。

## §5. 任務交接與協作流程

### 5.1 HANDOFF 鏈

HANDOFF 鏈是工作交接的真理。每個 Session 結束必須新增一份 `HANDOFF_N.md`，下個 Session 從上一份讀起。

**內容必含項目**（缺一不可，由 PM 撰寫、工程師抽驗）：

1. **里程碑摘要**：本 session 達成的高層目標（一段話）。

2. **完整膠囊清單**：列出**所有**完成的任務膠囊檔名（含 hotfix / 補登 / T0 探針等次膠囊），**禁止**為了敘事流暢把 hotfix 吞進主膠囊描述。每個 hotfix 都代表一段「真實技術債發現 → 修復」軌跡，下一個 session 必須能從 HANDOFF 看到這條軌跡。

3. **Protocols 版本迭代軌跡**：本 session 修改過的 `Dev_Protocol_IRON.md` / `Dev_Protocol_DISCIPLINE.md` / `PM_Operational_Manual.md` 的版號變化（如 `DISCIPLINE v1.2 → v1.7`），與該迭代涵蓋的關鍵條款（如「§7 工程師交付規範」「§5.2 接手義務」）。**不得只寫「制度有更新」這種模糊描述**。

4. **Institutional_Memory 新增段落引述**：本 session 寫入 `Institutional_Memory.md` 的新段落必須在 HANDOFF 摘要層級被點名（如「§S66-A BingX 冪等回應契約」「§S66-B-Hotfix SQLite Guid 大小寫陷阱」），並標註「未來何時要回頭讀」（如 SDK 升版、注入測試 SQL 時）。

5. **技術指標**：測試通過率、build 警告數、覆蓋的 VCP。

6. **下一階段預告**：明確的 NextWork 項目，與本 session 殘留的待辦（如 OS 校時、未補的 hotfix 等）。

**禁止項目**：

- ❌ **未經實證的描述性括號**：如「(已校準)」「(已測試)」「(已部署)」—— 若沒有對應 log / 檔案 / 使用者回報為證據，**禁止寫入**。改用「待校準」「未驗證」等誠實措辭，或標註佐證來源（如「已校準（使用者 2026-04-25 14:30 回報）」）。
- ❌ **把 hotfix 軌跡吞進主膠囊描述**：例如把 `TASK_S66B_HOTFIX_PERSISTENCE` 描述為「對帳服務 ... 排除 SQLite Guid 格式陷阱」這種把過程吞掉的寫法 —— 必須明確標出「兩張 hotfix 膠囊」並各自一行說明根因發現過程。
- ❌ **省略 protocols 版本迭代**：本 session 修過憲章的話必須紀錄；下個 session 工程師讀 HANDOFF 時必須能看到「協作協議升級」這條軸。

**抽驗權**：工程師收到 PM 完成的 HANDOFF 後**必須**抽驗一次內容真實性（與實際檔案、git log、testcase 數量比對），不符合即回頭要求 PM 補強。本抽驗權與 §5.2 條款 4「驗收測試計畫接手義務」同源 —— 工程師對 PM 產出的事實準確度有把關責任。

### 5.2 任務膠囊 (Capsules)
- **標準格式**：PM 產出任務膠囊必須嚴格遵守標準格式（目標、實作清單、VCP、交付要求）。
- **驗收紀錄回寫**：PM 必須將使用者的驗證結果或工程師提供的證明，**忠實且完整**地回寫到 Task 膠囊的最下方歷史紀錄區，以確保技術債與解決方案具備可追溯性。
- **指令交付**：PM 給予工程師任務時，需產出清晰具體的 Actionable Directive，並優先使用 DiagnosticTool 指令進行檢核。
- **驗收測試計畫接手義務**：收到工程師交付的「PM 驗收測試計畫」（見 §7）時，PM 必須**親自依序執行**所有非 🟡 可選情境，把每段終端機輸出**原文**貼回對話讓工程師判讀；**不得只回「收到」「明白」就視為驗收完成**，亦不得僅憑工程師單元測試的綠燈就自行宣告膠囊關閉。本義務由 §7 條款 0「Directive Header」觸發 — 看到 `📨 致 PM (Gemini)` 即為觸發信號。

### 5.3 驗證反饋 (VCP Loop)
驗收採用「語法交付模式」：提供純淨語法塊，指導使用者在地端執行並回寫閉環紀錄。

## §6. 雙 AI 協作實戰流水線 (Workflow Pipeline)

本章節定義了從需求開發到最終存檔的標準五階段流程。

### 6.1 階段一：任務契約化 (Task Templating)
- **行動**：PM 產出「任務膠囊 (Task Capsule)」。
- **規範**：膠囊必須包含「驗收檢核點 (VCP)」，預先定義用於驗收的 `DiagnosticTool` 指令輸出。
- **目標**：確保工程師在編碼前，就已知曉「何為正確」。

### 6.2 階段二：工程執行與單元測試 (Execution & Test)
- **行動**：工程師依據膠囊執行 `Plan -> Act -> Test`。
- **規範**：工程師僅限修改 `src/` 與 `tests/`；變更必須通過 `dotnet test` 並提供數據證明。

### 6.3 階段三：診斷先行驗收 (Diagnostic Verification)
- **PM 第一輪驗收**：PM 親自執行 `DiagnosticTool` 指令進行靜態與結構驗證，確保工具可用、路徑正確。
- **地端實機驗收**：PM 提供精確語法，請使用者在地端（具有 API Key/真實環境）執行，獲取實機 Log（如探針數據）。
- **數據回寫**：PM 將驗收 Log 忠實回寫至任務膠囊的「驗證反饋」區，達成閉環。

### 6.4 階段四：暫存區堆疊機制 (Context Accumulation)
- **檔案**：`management/DRAFT_CONTEXT.md`。
- **原則**：**「即時性與只增不刪」**。在 Session 期間，每完成一個原子步驟（如取得實測數據、測試通過、決策轉向），PM 必須**立即**更新交付摘要、VCP 紀錄、技術債與決策細節。
- **目標**：防止在對話過程中流失關鍵證據，並確保在對話異常中斷時，進度具備「物理存檔」。

### 6.5 階段五：存檔與交接 (Finalizing & Handoff)
- **觸發**：使用者執行 `/checkpoints save`。
- **行動**：PM 將 `DRAFT_CONTEXT.md` 的彙整內容寫入新的 `HANDOFF_N.md`、更新 `NextWork.md` 進度，並將完工膠囊歸檔至 `history/archive/`。
- **清理**：存檔完成後，PM 始可清空 `DRAFT_CONTEXT.md`。

## §7. 工程師完工交付規範 — PM 驗收測試計畫

**位階**：工程師（ClaudeCode）每完成一個開發任務（任意大小，包含 hotfix 與小修補，**不限於完整膠囊**），交付摘要的最後一節必須附上 `🧪 PM 驗收測試計畫`。

**理由**：PM (Gemini) 看不到實作細節、不修改 src/、無 shell 操作權，只能透過 DiagnosticTool / UI / 行行為觀察進行驗收。若工程師只交付「全綠」結論而不提供可執行的驗收路徑，PM 無法獨立判斷代碼是否真正解決問題，協作會退化成單方面信任。工程師對實際運行行為的掌握度高於 PM，**翻譯成 PM 可執行的測試步驟是工程師的責任**，不是 PM 的研究功課。

**結構要求**（缺一不可）：

0. **致 PM 執行指令（Directive Header，必置於計畫第一塊）**：明確要求 PM 接手執行測試而非只「接收」資訊。範本：
    > 📨 **致 PM (Gemini)**：以下測試計畫由工程師交付，請你接手執行 —— 工程師的 sandbox 無法跨 session 跑 ConsoleApp / sqlite3 / 真實 BingX。請**依序執行**情境 1 → 2 → ...，把每段終端機輸出**原文**貼回此對話。我（工程師）會判讀並決定是否關膠囊。可選情境（標 🟡）失敗不阻擋驗收。

    **理由**：使用者把工程師交付的計畫 forward 給 PM 時，沒有明確 directive 的話 PM 可能只「接收」（讀完不執行），協作循環卡住。明文要求「執行 + 原文回貼」是工程師對工作流轉的責任。

1. **前置檢查 (Pre-flight)**：列出測試開跑前必須確認的環境/資料狀態（Mode、餘額、金鑰、特定 DB 紀錄等）。**用具體 DiagnosticTool 指令驗證**，禁止只寫文字描述。

2. **測試情境 N**（建議 3-5 個，覆蓋主路徑 + 1-2 個邊界）：每個情境必須包含：
    - **目的**：本情境驗證的具體行為
    - **指令**：可一鍵複製貼上的 **純粹代碼區塊（嚴禁行號）**。
    - **期望輸出**：圈出關鍵字串/狀態碼錨點（如 `Status=New`、`✅ Path-A`），不接受「正常」這種模糊描述
    - **失敗解讀**：以表格列出「看到 X → 代表 Y → 修法 Z」，讓 PM 看到紅燈時能初步分流，不必每次回頭問工程師
    - **危險度標籤**（強制）：`📖 唯讀` / `⚠️ 會建立交易所端紀錄` / `🔥 會動到資金（嚴禁 Live）`

3. **UI / 視覺檢核**（若有）：DiagnosticTool 涵蓋不了的部分（Dashboard 顯示、廣播事件、Strategy Status 變化）必須寫明操作步驟與期望畫面元素，並標註「非阻塞 / 觀察項」與「核心驗收項」的差別。

4. **整體驗收判定**：明確列出「全綠 → 可關膠囊」「哪幾條紅 → 對應到哪個風險」的決策表。

**強制原則**：

- **任何寫真單情境必須附 Demo 模式檢查** —— 前置檢查必須驗證 `Mode=Demo`，並在情境步驟內第二次確認，雙保險避免 PM 在 Live 環境誤跑。
- **指令格式**：必須是純粹代碼區塊，嚴禁行號，確保一鍵複製可用。
- **不要包山包海** —— 每個情境必須能在 5 分鐘內判定綠/紅；過大的驗收計畫應拆膠囊。

**模板**：以 S66-A T0 探針補登（2026-04-25）的交付為標準範例；新進工程師可參照當日對話紀錄。
